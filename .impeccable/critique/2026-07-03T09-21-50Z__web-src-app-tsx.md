## Anti-Patterns Verdict

**LLM assessment:** The product mostly avoids obvious AI-dashboard tropes. Neutral color, dense tables, command access, and restrained motion match the intended calm finance tool. The strongest generic tell is composition: four equal metric cards, including one unavailable metric, followed by repeated bordered surfaces. The UI feels assembled from competent patterns but not fully edited around actual product readiness.

**Deterministic scan:** Three warnings. `styles.css:161` and `styles.css:1602` flag Arial as an overused font. More importantly, these declarations override the Inter stack declared on `:root`, creating a real typography contract mismatch. `styles.css:5256` flags width animation on budget progress bars. This is a low-impact detector hit because the element is small and reduced-motion handling exists, but `transform: scaleX()` would avoid layout work.

**Visual overlays:** Browser mutation and overlay tools were unavailable in this session. No user-visible overlay was created. Assessment used source, styles, component tests, and the deterministic detector.

## Overall Impression

Trustworthy foundation, unfinished editing. Biggest opportunity: make every visible control and summary truthful about available data and implemented capability.

## What's Working

- The app shell has clear navigation, persistent collapse state, global command access, and a separate transaction search model.
- Financial data uses explicit signs, currencies, tabular numbers, and text labels instead of relying only on color.
- Dialogs, focus restoration, keyboard shortcuts, reduced-motion handling, and localized copy show unusually good accessibility intent.

## Priority Issues

### [P1] Transaction history search is incomplete but looks complete

**Why it matters:** `TransactionsView` and `TransactionSearchDialog` call `api.transactions()` once, then filter locally. The API now defaults to 50 rows. Older records silently disappear from filters and search, which breaks the product promise of an auditable ledger.

**Fix:** Move search/filter/date/page state into the API query, expose result count and pagination, and make command search clearly scoped if it only searches recent records.

**Suggested command:** `/impeccable harden`

### [P1] Prime UI space exposes unavailable features

**Why it matters:** The dashboard shows a subscriptions metric as zero plus “Unavailable”; the topbar exposes an empty-state preview that only produces a toast; import opens a placeholder. Users learn that prominent actions may be fake.

**Fix:** Remove unavailable actions from primary navigation. Put future features behind one quiet “Coming later” area or ship a complete thin slice before exposing them.

**Suggested command:** `/impeccable distill`

### [P1] Loading and failure states erase useful structure

**Why it matters:** Dashboard and transaction loading replace the task surface with plain `Empty` text. Errors lack direct retry. This creates a visual cliff on slow/self-hosted networks and weakens confidence in a finance tool.

**Fix:** Use existing Skeleton components to preserve layout, keep stale data visible during refetch, add inline retry actions, and distinguish empty, loading, offline, and server-error states.

**Suggested command:** `/impeccable harden`

### [P2] Too many competing top-level actions

**Why it matters:** Sidebar toggle, command, transaction search, categories, empty-state preview, create account, rail toggle, and notifications compete in one header. This exceeds the four-item working-memory threshold before page-specific actions appear.

**Fix:** Keep command, search, and one context-primary action. Move categories into settings or command menu; remove dead preview; group low-frequency utilities in an overflow menu.

**Suggested command:** `/impeccable distill`

### [P2] Visual vocabulary is internally inconsistent

**Why it matters:** `shared/ui` wrappers and direct shadcn imports coexist in feature files; `:root` declares Inter while `body` and auth force Arial. Small differences in focus, spacing, select behavior, and typography accumulate into a subtly untrustworthy product feel.

**Fix:** Pick one component entrypoint, remove legacy variants, define one font stack token, and standardize loading/error/empty component states.

**Suggested command:** `/impeccable extract`

## Persona Red Flags

**Alex, power user:** Search and filters only cover the loaded transaction slice, so old records cannot be found reliably. Five filters are always visible, but pagination and result count are absent. Fast-looking controls produce incomplete answers.

**Sam, keyboard and screen-reader user:** Shortcut and dialog foundations are strong. Risks remain in mixed component implementations, loading text without announced busy regions, errors without consistent `role="alert"`, and muted 11–12px labels that need contrast verification.

**Casey, distracted mobile user:** The topbar carries too many actions before page content. Important create actions remain near the top, while interrupted dialog forms have no explicit draft persistence. Dense filter controls require substantial vertical scanning on narrow screens.

## Minor Observations

- The fourth dashboard metric should not be an unavailable module.
- Weekly cashflow offers a selectable state the backend cannot provide; hide it until supported.
- Version appears both in sidebar health and dashboard heading, adding operational detail twice.
- Progress width animation is minor but can use `scaleX`.
- Several 11–12px muted labels need measured WCAG contrast in both themes.

## Questions to Consider

- Is CapitalFlow a complete daily ledger first, or a preview of future finance modules?
- Should global search mean all history, or explicitly “recent operations”?
- Which single action should dominate the topbar on each page?
